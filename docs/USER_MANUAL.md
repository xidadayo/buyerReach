# BuyerReach 买手通使用手册

## 本地邮箱验证

在“系统配置 → Provider”中找到“本地邮箱验证”：

1. 填入部署环境 `VERIFIER_TOKEN` 对应的内部 Token 并保存。
2. 点击连接测试；该测试不验证真实邮箱，也不消耗第三方额度。
3. 首次上线选择“影子对比”，流量设为100%，保持付费结果作为最终结果。
4. 对比稳定后选择“正式启用”，按10%、30%、60%、100%逐步放量。
5. 本地 Unknown、Catch-all或临时错误会自动交给ZeroBounce/Hunter。

回滚时将本地验证模式设为“禁用”或关闭该Vendor；新任务立即恢复原验证流程，历史记录不会删除。

适用版本：V1.0  
适用对象：系统管理员、数据管理员、业务人员、审核人员

BuyerReach 用于沉淀品牌、联系人、邮箱、来源证据和验证结果。它不是邮件群发、完整 CRM 或 LinkedIn 自动化抓取工具。

## 1. 访问系统

默认地址：

| 服务 | 地址 | 用途 |
| --- | --- | --- |
| 管理后台 | `http://localhost:5173` | 日常操作入口 |
| API 文档 | `http://localhost:8000/docs` | 管理员与集成开发使用 |
| 后端健康检查 | `http://localhost:8000/health` | 检查服务是否已启动 |
| 后端就绪检查 | `http://localhost:8000/ready` | 检查数据库连接是否正常 |
| MinIO Console | `http://localhost:9001` | 对象存储运维入口 |
| n8n | `http://localhost:15678` | 外部流程编排入口 |

首次启动会创建开发环境管理员：

```text
账号：admin@buyerreach.local
密码：admin123
```

首次登录后请在右上角用户菜单中选择“修改密码”。生产环境必须使用非默认的 `JWT_SECRET` 与 `ENCRYPTION_KEY`；系统会拒绝使用默认密钥启动生产环境。

## 2. 启动、停止与更新

### Windows + WSL Docker

在项目根目录的 PowerShell 中执行：

```powershell
wsl.exe sh -lc "cd '/mnt/d/buyer reach' && docker compose -f docker-compose.yml -f docker-compose.wsl.yml --env-file .env.example up -d --no-build --pull never"
```

修改前端、后端或依赖后，使用以下命令重建：

```powershell
wsl.exe sh -lc "cd '/mnt/d/buyer reach' && docker compose -f docker-compose.yml -f docker-compose.wsl.yml --env-file .env.example up -d --build backend celery-worker celery-beat frontend"
```

查看运行状态：

```powershell
wsl.exe sh -lc "cd '/mnt/d/buyer reach' && docker compose -f docker-compose.yml -f docker-compose.wsl.yml --env-file .env.example ps"
```

停止服务但保留数据：

```powershell
wsl.exe sh -lc "cd '/mnt/d/buyer reach' && docker compose -f docker-compose.yml -f docker-compose.wsl.yml --env-file .env.example down"
```

### Linux/NAS

安装 Docker 与 Docker Compose 后，复制项目与环境变量文件，在项目根目录执行：

```sh
cp .env.example .env
# 编辑 .env，填入生产数据库密码、JWT_SECRET、ENCRYPTION_KEY 和 Provider 密钥
docker compose up -d --build
```

不要在生产环境直接使用 `.env.example` 的默认密钥或密码。

## 3. 角色与权限

系统使用 RBAC 权限控制。预置角色如下：

| 角色 | 典型职责 |
| --- | --- |
| `admin` | 用户、角色、配置、Provider、审计、导入导出与所有业务操作 |
| `operator` | 品牌、联系人、邮箱、任务、导入导出与去重操作 |
| `viewer` | 只读查看品牌、联系人、邮箱和任务 |

管理员可进入“系统配置 → 用户与角色”：

1. 在“角色与权限”中新建角色或编辑权限 JSON。
2. 在“用户”中新建用户，设置姓名、邮箱、初始密码、角色和状态。
3. 停用用户会阻止其继续通过 Token 访问系统。
4. 修改用户密码时，留空“重置密码”即可不修改原密码。

权限 JSON 示例：

```json
{
  "brands": ["read", "write"],
  "contacts": ["read"],
  "emails": ["read", "verify"],
  "tasks": ["read", "write", "execute"]
}
```

## 4. 首次配置

在创建搜索任务前，请先进入“系统配置”。配置按以下顺序进行。

### 4.1 业务规则

“业务规则”页用于维护：

- P1、P2、P3 目标职位词库；每行一项。
- 排除职位词库，例如 `Intern`、`Recruiter`。
- 有效邮箱与风险邮箱的评分阈值。
- 任务最大重试次数、重试等待时间和最大并发。
- 每个品牌默认联系人数量。

保存后，邮箱验证会按“有效邮箱评分阈值”决定是否将结果放入有效邮箱池；评分不足的 `valid` 结果会进入风险/人工审核流程。

### 4.2 Provider 配置

Provider 是公司搜索、联系人搜索、邮箱发现、邮箱验证和通知的统一接入层。前端不会直接访问第三方服务。

#### 多源瀑布与获取流程

同一类型可以配置多个 Provider。系统会按“优先级”从小到大依次尝试：主来源额度耗尽、请求失败或没有映射结果时，自动切换至下一个已启用来源；邮箱验证在返回 `unknown` 时也会继续尝试备用验证服务。失败和跳过记录会显示在“搜索任务 → 详情”中。

建议配置顺序：

1. `company_search`：企业目录/商业数据源作为主来源，官网或其他数据源作为备用。
2. `contact_search`：联系人数据源按目标市场和职位覆盖率排序。
3. `email_finder`：邮箱发现服务优先，系统在均无结果时再使用姓名与域名的模式推断。
4. `email_verifier`：配置至少两个验证来源，最终将有效邮箱放入有效池，风险和未知邮箱进入人工审核池。

“每月额度”按自然月累计调用次数计算；额度填空表示不限制。请先点击“测试”确认每个 Provider 的认证、字段映射和返回路径正确，再启用。

新增 Provider 的步骤：

1. 打开“系统配置 → Provider”，选择“新增 Provider”。
2. 填写名称与类型：企业/品牌搜索、联系人搜索、邮箱发现、邮箱验证或通知。
3. 选择适配器：
   - **内置**：适合演示、种子数据和本地流程验证。
   - **HTTP**：连接正式第三方 API 或 Webhook。
4. 设置优先级；数字越小，越先被调用。
5. 设置每月额度，并按需要启用该 Provider。
6. 保存后点击“测试”，确认返回成功。

HTTP Provider 常用字段：

| 字段 | 说明 |
| --- | --- |
| 接口 URL | Provider 的完整 API 地址 |
| HTTP 方法 | `POST` 或 `GET` |
| API Key / 认证 Header | 第三方认证信息；系统会加密保存并在列表中脱敏显示 |
| 响应根路径 | 第三方响应中实际业务对象所在路径 |
| 数据列表路径 | 公司、联系人或邮箱数组所在路径 |
| 字段映射 | 将第三方字段映射为 BuyerReach 标准字段的 JSON |

#### 关键词品牌发现

使用 HTTP `company_search` Provider 接入搜索引擎或公开目录时，在 Provider 配置中选择“搜索引擎”或“公开目录”。接口 URL 支持 `{{query}}`、`{{brand_keywords}}`、`{{categories}}`、`{{countries}}` 占位符；`POST` 请求也会收到组合后的 `query` 字段。

搜索结果字段映射建议如下：

```json
{
  "brand_name": "title",
  "website": "link",
  "country": "country",
  "category": "category",
  "source_url": "link",
  "source_title": "title",
  "source_excerpt": "snippet"
}
```

创建“品牌发现”任务时，填写名词/品类、国家和最低相关性。系统只保存达到阈值且符合官网要求的候选品牌，并记录相关性分数和来源证据；不会直接发起联系人与邮箱开发。到“品牌库”点击候选品牌的“批准并丰富”后，系统会创建精确品牌丰富任务，继续执行官网、联系人、邮箱和验证流程。

内置公司搜索可填入“品牌种子 JSON”，示例：

```json
[
  {
    "brand_name": "Example Brand",
    "website": "https://example.com",
    "country": "US",
    "category": "handbags"
  }
]
```

通知 Provider 使用 HTTP 适配器。启用后，任务完成、失败、去重、导入导出等事件会向它发送事件名和业务载荷。也可通过环境变量 `FEISHU_WEBHOOK_URL` 使用内置飞书通知。

### 4.3 标签与自定义字段

在“系统配置 → 字段与标签”中：

- 新增品牌、联系人或邮箱标签，用于数据分类。
- 新增文本、数字、日期、单选、多选、布尔、URL、邮箱或电话字段。
- 设置字段是否必填、可筛选和在列表中展示。

### 4.4 审计日志

“系统配置 → 审计日志”可查看登录、配置变更、导入导出、任务、验证、去重和黑名单操作。管理员应定期检查异常导出、权限修改和失败任务。

## 5. 日常业务流程

推荐流程：

```text
配置 Provider → 创建搜索任务 → 查看品牌/联系人/邮箱 → 验证与审核
→ 去重/黑名单 → 导出有效数据
```

### 5.1 创建并执行搜索任务

进入“搜索任务”：

1. 点击“新建任务”。
2. 选择“品牌发现”或“精确品牌”。
3. 品牌发现任务不填写品牌名，只填写品牌归属国家、目标品类、最低相关性、目标职位和品牌上限；系统据此发现未知品牌。精确品牌任务才填写已知品牌名和已确认的官方域名。
4. 保存后，任务状态为草稿。
5. 点击“启动”，任务会进入 Celery 后台队列；关闭浏览器不会中断任务。
6. 列表会轮询显示品牌、联系人、邮箱数量和错误信息。

任务状态说明：

| 状态 | 含义与操作 |
| --- | --- |
| 草稿 | 可启动、取消或复制 |
| 排队中 / 执行中 | 可暂停或取消 |
| 已暂停 | 可继续，继续后会重新入队 |
| 已完成 | 可查看详情或复制为新草稿 |
| 失败 | 查看错误后重试失败项或复制任务 |
| 已取消 | 保留历史记录；如需重跑请复制 |

点击“详情”可查看筛选条件、错误信息及各阶段任务明细。暂停或取消会在当前品牌/联系人处理边界生效，已经保存的数据不会被删除。

### 5.2 品牌库与官网解析

在“品牌库”中可以：

- 手工新增品牌、公司、官网、国家和品类。
- 查看品牌发现候选的相关性分数和来源证据；确认后点击“批准并丰富”。
- 编辑品牌信息或归档不再使用的品牌。
- 对有官网地址的品牌点击“解析官网”。

官网解析会提取公开邮箱、电话、社交媒体链接、标题、页面摘要和来源证据。解析结果中的公开邮箱会保存并进入验证流程；解析失败时可查看错误原因后重试。

### 5.2.1 品牌候选池：匹配度评估与批量精准丰富

“品牌候选池”回答的核心问题是：**这条候选和我搜索的关键词匹配吗？值不值得付费精准丰富？**

- 页面顶部可按“搜索任务”筛选。选中任务后，页面显示该任务的名称、搜索目标、目标概念和筛选条件；列表展示的是**该任务自己的匹配结果**，后续搜索不会覆盖历史任务的评估。
- “相关度”列只展示正式匹配结果：
  - 已完成评估：A/B/C/D 评级徽章 + 正式百分比分数（A ≥ 85，B ≥ 65，C ≥ 40，D < 40）。
  - 其他状态显示“待评估 / 评估中 / 证据不足 / 评估失败，可重试”，不会显示估算百分比。
- 点击行首展开按钮查看匹配详情：命中的目标概念、未覆盖的目标概念、冲突概念、六维评分明细（产品 40 / 行业 20 / 商业类型 15 / 国家 10 / 证据质量 10 / 品类覆盖 5）、判定原因、证据来源、评分版本和推荐操作。详情中的“资料完整度 X/7 项”只表示资料覆盖程度，不是匹配度。
- 列表支持按正式匹配分或最近发现排序（未评估候选始终排在有正式分数的候选之后），并按评级、评估状态筛选。
- 行业补充失败的候选会显示简短可操作的失败原因（官网不可访问 / AI 或网络异常 / Provider 配额耗尽 / Provider 未配置 / 证据不足）；配额耗尽且 Provider 提供重置时间时会显示重置时间。点击“重试补充”可重新发起；排队或进行中时按钮禁用，防止重复提交。
- 批量精准丰富：点击“选择本页 A/B 候选”快速选中合格候选，再点“批量精准丰富”。服务端逐条校验（仅已完成正式评估且评级 A/B 的候选会真正发起），每条独立创建任务，一条失败不影响其他；完成后展示逐条成功、跳过、失败明细，未成功的候选保持选中便于处理。重复提交不会创建重复任务。

### 5.3 联系人与邮箱

在“联系人”页新增联系人时，选择所属品牌并填写姓名、职位和 LinkedIn 资料链接。LinkedIn 仅作为人工核实入口，系统不会自动登录或批量抓取。

在“邮箱池”中可以：

- 手工新增个人或通用邮箱，并选择关联联系人。
- 点击“验证”重新执行邮箱验证。
- 对人工审核池中的邮箱执行通过、拒绝或禁止发送。

邮箱池含义：

| 邮箱池 | 建议处理 |
| --- | --- |
| 原始池 | 新发现、尚未验证的邮箱 |
| 待验证池 | 已排队等待验证 |
| 人工审核池 | 风险、未知或评分不足，需要人工判断 |
| 有效邮箱池 | 满足验证和评分规则，可进入后续业务流程 |
| 无效邮箱池 | 格式、域名或 Provider 明确判定无效 |
| 禁止发送池 | 黑名单、退订或人工禁止；不得再次进入有效流程 |

## 6. 审核、去重、导入和导出

进入“审核与去重”：

### 6.1 导入

支持 CSV 和 XLSX。操作步骤：

1. 选择品牌、联系人或邮箱数据类型。
2. 上传文件后点击“预览”，确认表头和前 20 行数据。
3. 如表头不符合标准字段，填写字段映射 JSON。键为系统字段，值为源文件列名。
4. 点击“开始导入”，查看成功、跳过和错误数量。

品牌映射示例：

```json
{
  "name": "Brand Name",
  "company_name": "Company",
  "website": "Website",
  "country": "Country",
  "category": "Category"
}
```

常用标准字段：

| 数据类型 | 字段 |
| --- | --- |
| 品牌 | `name`、`company_name`、`website`、`country`、`category` |
| 联系人 | `brand_id`、`company_id`、`first_name`、`last_name`、`title`、`linkedin_url` |
| 邮箱 | `contact_id`、`address`、`type` |

### 6.2 去重与合并

点击“执行检查”后，系统展示精确匹配和模糊匹配候选。合并前请确认主记录正确：合并会保留关联记录与历史，并将重复记录软删除。

### 6.3 禁止发送名单

可按邮箱或域名新增禁止发送项。已在名单中的邮箱验证时会自动进入禁止发送池，优先级高于其他验证结果。

### 6.4 导出

选择需要导出的品牌、联系人、邮箱、任务或审计日志。可选填写一个精确筛选字段和值，例如：

```text
筛选字段：status
筛选值：valid
```

导出文件为 UTF-8 with BOM 的 CSV，可直接用 Excel 打开。完整邮箱导出应仅授予已获授权的角色，并定期审计导出记录。

## 7. 备份与恢复

备份会包含 PostgreSQL 数据库、MinIO 数据和环境配置，并使用 AES-256-CBC 加密。

### 创建备份

在 Linux/NAS 的项目根目录执行：

```sh
BACKUP_PASSPHRASE='请使用密码管理器保存的强口令' ./scripts/backup.sh
```

输出文件位于 `backups/`，扩展名为 `.tar.gz.enc`。建议每日执行，并将备份复制到独立的受保护存储。

### 恢复备份

恢复会覆盖目标数据库和 MinIO 数据，请务必在独立测试环境演练后再用于生产：

```sh
CONFIRM_RESTORE=YES BACKUP_PASSPHRASE='请使用密码管理器保存的强口令' \
  ./scripts/restore.sh backups/buyerreach-<timestamp>.tar.gz.enc
```

恢复完成后检查：

1. `http://<host>:8000/ready` 返回 `ready`。
2. 管理后台能登录并显示品牌、联系人和邮箱数量。
3. MinIO 数据可访问。
4. Worker 和 Beat 容器处于运行状态。

## 8. 常见问题

### 登录返回 401

- 确认账号、密码和服务地址正确。
- 初始账号仅适用于首次开发环境启动；如已改过密码，请使用新密码。
- 连续 5 次失败会锁定账号 15 分钟，请等待或由管理员重置密码。

### 启动任务提示没有 Provider

进入“系统配置 → Provider”，至少新增并启用一个 `company_search` Provider。若需联系人、邮箱和验证完整流程，还需启用对应类型 Provider。

### 任务失败

在“搜索任务 → 详情”查看 Provider、阶段和错误信息。修正 Provider 地址、认证、额度或字段映射后，使用“重试失败项”或复制任务重新运行。

### Provider 测试失败

- 检查 URL、HTTP 方法、认证 Header 和 API Key。
- 检查响应根路径、数据列表路径和字段映射是否对应第三方返回结构。
- 检查服务器是否能访问第三方 API，以及额度是否耗尽。

### 官网解析不到邮箱

确认官网可从服务器访问，且邮箱确实公开出现在页面中。部分站点使用图片、动态脚本、联系表单或反爬策略，可能无法提取公开邮箱。

### `/ready` 失败

通常表示后端无法连接 PostgreSQL。执行 `docker compose ps` 和 `docker compose logs backend postgres`，确认数据库处于健康状态且环境变量中的 `DATABASE_URL` 正确。

## 9. 运维检查清单

- 每日：检查失败任务、Provider 额度、人工审核池和备份结果。
- 每周：检查审计日志、黑名单、无效邮箱比例和用户权限。
- 每月：恢复一份备份到独立环境，验证数据库、MinIO 和登录功能。
- 发布前：备份数据，执行迁移，检查 `/health` 与 `/ready`，完成一次任务、导入、验证和导出冒烟测试。
