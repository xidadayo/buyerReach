# BuyerReach 后续版本开发路线图（V1.5–V3.0）

总方案入口：`docs/buyerreach-final-development-technical-plan-v1.md`。

状态：架构设计基线
日期：2026-07-21
需求来源：`BuyerReach_完整产品需求与版本开发规范_V1.0-V3.0.docx`
当前实现基线：`docs/production-integrity-audit-2026-07-17.md`、Pipeline V2、查询切片生产方案

## 1. 路线图决策

保留原产品文档的四级业务方向，但不将每一级作为一次大爆炸式发布：

- V1.5：智能筛选、持续数据供给与质量维护；
- V2.0：合规、可控、可审计的邮件开发；
- V2.5：销售协同、询盘、样品和报价闭环；
- V3.0：品牌监控、采购信号和智能推荐。

每个主版本拆成可以独立发布、验证和回滚的纵向子版本。后续版本只能在前置版本的发布门通过后启用，不允许以仅存在数据表或页面代替生产闭环。

当前下一版本不是直接开发邮件群发，而是完成 V1.5 的数据供给与数据质量闭环。原因是开发信、CRM 和采购信号都依赖持续更新、可追溯、经过筛选的公司与联系人数据。

## 2. 当前能力与原需求差距

### 2.1 已具备或部分具备

- 公司、品牌、联系人、邮箱、来源证据和审核基础；
- Provider Adapter、Vendor Plan、额度检查和失败切换；
- Pipeline V2、持久 Stage Run、状态机、幂等、Lease 和恢复扫描；
- Outbox、数据库事件和认证 SSE；
- AI/规则分离的相关性判断与评分历史；
- Candidate 行业丰富、概念匹配和人工审核；
- API 调用量、成本基础记录；
- 任务暂停、恢复、取消、部分结果和旧任务兼容；
- 本地/第三方邮箱验证瀑布及风险分类基础。

### 2.2 V1.5 关键缺口

- 查询切片计划尚未正式保存并驱动 Provider 执行；
- 缺少可持续的增量发现、来源级游标和重复执行策略；
- 缺少正式的客户画像模板；
- 缺少统一数据新鲜度、复查计划和变化记录；
- 缺少面向业务人员的数据质量中心与修复队列；
- Recompute 持久模型存在，但用户可用的批次创建、暂停、恢复和对比尚未开放；
- 成本记录存在，但有效候选/合格客户获取成本尚未形成可解释看板；
- 自动标签和规则生命周期尚未形成完整闭环。

### 2.3 V2.0 及以后尚未形成的领域

- 发送账户与发送域名健康；
- 模板审批与版本；
- 邮件序列、活动和发送调度；
- 全局/品牌/域名抑制策略的正式 Outreach 边界；
- 送达、退信、回复、退订和投诉事件；
- CRM 线索、商机、活动、询盘、样品、报价和审批；
- 页面监控、变化事件、采购信号、推荐和知识库。

## 4. V1.5：智能筛选与持续数据维护

### 4.1 V1.5A 查询切片与持续发现

目标：一个用户动作启动可审核、可锁定、可重复执行的多切片搜索，并在多任务并发时稳定运行。

权威实施方案：

- `docs/query-slicing-production-implementation-plan.md`；
- `docs/query-slicing-production-implementation-prompt.md`；
- `docs/query-slicing-implementation-status.md`。

交付：

- Query Plan、Slice、SliceRun；
- 目标合格量、候选上限、预算和停止条件；
- 单页小步调度、公平准入、容量 Lease；
- 切片级游标、耗尽、重试和来源命中；
- 现有 Provider 的能力路由与 fallback；
- 单页快速创建和可选切片抽屉；
- SSE 任务中心和明显的排队/部分结果体验；
- `new_only` 重复执行基础；
- disabled/shadow/review/active rollout。

发布门：

- 普通用户最多三次操作启动；
- 切片真实驱动 Provider，不再只是 Stage 输出；
- 多任务公平、可恢复且不拖垮交互 API；
- 重复执行默认不重抓相同页；
- 迁移、回滚、并发、故障和视觉验证通过。

### 4.2 V1.5B 数据新鲜度与质量中心

目标：从“发现一次”升级为“持续维护可用数据”。

主要用户流程：

```text
系统识别过期/缺失/冲突数据
-> 自动创建复查批次
-> 按预算和优先级执行
-> 展示字段变化与证据
-> 确定性变化自动应用或进入人工审核
-> 更新新鲜度和下一次复查时间
```

建议实体：

- `review_policy`：对象类型、复查周期、优先级、预算和适用范围；
- `review_schedule`：entity、policy_version、next_run_at、状态；
- `data_observation`：来源对字段的时间序列观测；
- `data_change`：old/new、来源、置信度、应用状态；
- `quality_issue`：缺失、冲突、过期、无来源、失败和责任人；
- `quality_batch`：可暂停、恢复、限速和重试的批次。

业务规则：

- `last_seen_at` 不等于 `last_verified_at`；
- 每个来源独立记录观察时间；
- 证据冲突不静默覆盖，经字段政策判断自动应用或审核；
- 复查任务必须幂等并受预算、取消和并发控制；
- 联系人离职、邮箱失效、官网变化分别使用独立状态和证据；
- 删除或无法访问不直接等同于公司消失。

质量中心至少展示：

- 数据年龄分布；
- 缺失、冲突、过期和无来源数量；
- 来源覆盖率；
- 复查成功率、失败原因和成本；
- 待人工处理队列；
- 自动修复与人工覆盖历史。

发布门：

- 同一复查重复投递不重复产生变化记录；
- 旧数据、冲突、失败和取消路径完整；
- 质量指标来自实际数据，不使用估算值冒充；
- 批次可以暂停、恢复、限速和回滚自动应用策略。

### 4.3 V1.5C 客户画像、重算、标签与成本

目标：将查询、评分、标签和成本统一在可复用客户画像下。

建议实体：

- `lead_profile` 与不可变 `lead_profile_version`；
- `profile_query_plan_binding`；
- `tag_rule` 与 `tag_rule_version`；
- `tag_assignment_history`；
- 复用并开放 `RecomputeBatch` 和 `RelevanceScoreHistory`；
- `cost_allocation`：将 Provider/AI/抓取成本归因到任务、切片和结果。

用户流程：

```text
保存客户画像模板
-> 一键创建查询计划
-> 查看画像覆盖和结果分布
-> 修改规则时选择 policy_only 或 AI 重算
-> 对比旧/新评分和标签
-> 审核后发布新策略版本
```

规则：

- AI 输出维度和证据判断，确定性 Policy 输出业务评分；
- `policy_only` 必须复用历史 AI 维度，不能再次调用 AI；
- 重算新增历史，不覆盖正式评分；
- shadow 评分不能改变正式业务决策；
- 自动标签必须记录规则版本、触发证据和撤销历史；
- 成本以真实调用和使用量计算，不承诺未经测量的节省比例。

发布门：

- 画像版本不可变，任务快照引用明确；
- 批次暂停/恢复、预算、对比和回滚通过；
- 标签规则不会覆盖人工禁止或人工固定标签；
- 成本汇总与底层使用记录可核对。

### 4.4 V1.5D 自动化控制平面

目标：建立统一的自动任务和定时任务基础，先服务定期增量发现与数据/邮箱复查，再在 V2.0 承载邮件发送和跟进。

权威架构：`docs/customer-development-automation-architecture-v1.md`。

多用户与权限权威架构：`docs/multi-user-access-control-architecture-v1.md`。

交付：

- Automation Definition、不可变 Version、Trigger、Run、StepRun 和 Approval；
- PostgreSQL 保存用户日程、时区、next_fire_at、状态和业务快照；
- Celery Beat 仅运行固定到期扫描器，不为每个用户计划动态建 Beat 条目；
- 时区、DST、misfire、幂等触发、Lease、暂停、恢复、预算和公平调度；
- 自动化中心、下一次运行、部分结果、等待审批和异常提示；
- Organization Membership、权限目录、数据范围和统一 AuthorizationContext；
- Automation Principal、权限快照与高风险步骤实时复核；
- 第一批真实消费者：定期增量发现、官网/邮箱复查；
- 不在本阶段发送真实客户邮件。

发布门：

- 重复扫描和重复事件只创建一个 Automation Run；
- 停机跨过执行时间后按锁定 misfire 策略补跑或跳过；
- 多组织公平，暂停后不产生新步骤；
- Worker/Redis 重启可从数据库恢复；
- 自动化版本升级不改变已经创建的 Run；
- 用户清晰看到下一次运行、当前阶段、部分结果和失败原因。
- 不同组织无法通过列表、详情、事件流、导出或对象存储访问彼此数据；
- 权限撤销、用户停用和自动化创建者离职都有确定的暂停、认领或恢复路径。

## 5. V2.0：合规开发信平台

V2.0 必须按“安全基础 -> 受控发送 -> 事件和回复”分三次发布。不得先做群发页面再补抑制和幂等。

### 5.1 V2.0A Outreach 安全基础

目标：建立尚不发送邮件的账户、域名、模板和抑制基础。

交付：

- Sending Account 与版本化 Provider Adapter；
- Sending Domain、DNS/认证检查和健康状态；
- Template、Template Version、语言和审批；
- 全局、组织、品牌、域名和邮箱范围的 Suppression；
- 发送政策、每日额度、时区和窗口；
- 模板变量白名单、预览和测试渲染；
- 权限、密钥治理和审计；
- 发送前风险检查 API。

安全不变量：

- 抑制名单优先级高于活动、序列、人工操作和 AI 建议；
- 凭证只经系统配置治理并加密，不进入任务/消息快照；
- 模板和政策锁定版本后不可被运行中活动改变；
- 默认只允许预览和测试地址，不对真实客户发送。

发布门：

- 无任何路径可绕过 suppression；
- 模板变量、权限、密钥、测试发送和审计通过；
- 账户或域名 unhealthy 时不能进入 ready。

### 5.2 V2.0B 活动、序列与受控发送

目标：将已审核客户池转为可暂停、可恢复、严格限速的邮件活动。

建议实体：

- `email_sequence`、`sequence_version`、`sequence_step`；
- `campaign`、`campaign_configuration_snapshot`；
- `campaign_recipient`；
- `email_message` 与唯一发送幂等键；
- `sending_slot_lease` 与账户/域名/品牌频控。

最小工作单元：一封消息。发送前必须再次检查活动、收件人、账户、域名、全局抑制、品牌频控、预算和窗口。

活动状态：

```text
draft -> validating -> ready -> running
running -> paused/completed/cancelled
validation failure -> draft or blocked
```

发布门：

- 重复队列投递最多产生一次实际发送；
- 暂停/取消后不再产生新发送；
- 账户、域名、品牌和全局限速不超配；
- 多账户公平调度；
- 活动配置和收件人池可追溯；
- 小比例内部测试、shadow 和人工批准完成。

### 5.3 V2.0C 事件、回复与效果分析

目标：可靠接收邮件事件并形成可操作回复队列。

交付：

- Webhook Adapter、签名验证、重放保护和事件幂等；
- delivered、soft/hard bounce、replied、unsubscribed、complained；
- 回复原文受控存储、线程关联和意图分类；
- positive/neutral/negative/referral/out_of_office/unsubscribe/unknown；
- 低置信度审核；
- 退订、投诉和硬退信自动进入 suppression；
- 按账户、活动、模板、国家和职位的效果分析。

不得将打开率作为核心成功指标；隐私保护可能使打开事件不可靠。核心指标优先使用送达、硬退信、回复、积极回复、退订和投诉。

发布门：

- Webhook 重复、乱序和延迟到达安全；
- 退订/投诉在同一事务生效并阻止后续调度；
- 回复分类不可用时进入 unknown/review，不自动创建积极商机；
- 历史消息和事件可重放构建状态。

## 6. V2.5：CRM 销售协同

### 6.1 V2.5A 线索、负责人、任务与时间线

目标：把积极回复和人工选择的品牌转成明确负责人和下一步行动。

交付：

- Lead、Assignment Rule、Owner History；
- Activity/Task、提醒和逾期；
- 统一 CRM Timeline，只保存事件引用和必要快照；
- 邮件、回复、备注、任务、文件和状态变化关联；
- 部门数据范围和转派审计；
- 飞书通知，不把飞书作为状态来源。

发布门：

- 重复回复不重复创建线索；
- 转派保留历史；
- 时间线可按数据库事件重建；
- 数据范围权限覆盖 API、搜索、导出和通知。

### 6.2 V2.5B 商机、询盘、样品、报价与审批

目标：形成从客户需求到样品和报价结果的业务闭环。

交付：

- Pipeline 与不可变 Stage Version；
- Opportunity 和阶段转换规则；
- Inquiry、结构化需求和附件；
- Sample、物流、费用、反馈和结果；
- Quotation Version、币种、Incoterm、有效期和附件；
- Approval、审批政策和超时；
- 管理报表和阶段 SLA。

规则：

- 阶段变化统一 transition，校验必填字段和权限；
- 报价新版本不覆盖历史；
- 金额、汇率和概率来源可追溯；
- 附件进入受控对象存储，下载鉴权并有审计；
- 飞书审批结果通过幂等 Adapter 回写，数据库仍为真相来源。

发布门：

- 阶段状态机、并发编辑、审批重复回调和附件权限通过；
- 商机、样品和报价历史完整；
- 回滚不删除业务记录。

## 7. V3.0：智能获客与采购信号

### 7.1 V3.0A 品牌监控与变化事件

目标：低频、许可合规地监控已批准目标，而不是构建无边界爬虫。

交付：

- Monitoring Target、Source Policy 和 Frequency；
- Page Snapshot 元数据与对象存储；
- 内容哈希、条件请求和变化检测；
- Change Event、before/after 引用、置信度和审核；
- robots、SSRF、限速、超时、响应大小和保留策略；
- 网站级并发与熔断；
- 新品、分类、门店、新闻、招聘等版本化 Extractor。

发布门：

- 重复快照不产生重复事件；
- 内容噪声不会批量制造信号；
- 快照存储与业务库分离；
- 删除、归档、许可和来源追踪完整。

### 7.2 V3.0B 采购信号与智能建议

目标：将经过验证的变化转成可解释、可审核的联系时机和建议动作。

交付：

- Signal Policy、Signal、Evidence；
- 新品、扩品类、市场扩张、融资、招聘和展会等信号；
- 确定性规则评分与 AI 摘要分离；
- 联系人优先级、产品切入点和建议动作；
- Alert Rule、去重、冷却期和负责人通知；
- 人工反馈和业务结果关联。

AI 不能直接发送邮件、改变商机阶段或绕过人工审批。低置信度信号进入审核。

发布门：

- 每个信号有来源、证据、版本和时间；
- 重复/相近信号合并且有冷却；
- shadow 评估不会创建正式机会；
- 人工反馈可追踪但不直接在线改变正式模型。

### 7.3 V3.0C 推荐、市场分析与管理驾驶舱

目标：在拥有真实回复、商机、样品、报价和转化数据后提供相似客户与市场建议。

交付：

- Similarity Profile 和版本化特征；
- 相似客户推荐及理由；
- 按国家、品类、渠道、职位、来源和信号的转化分析；
- 管理驾驶舱；
- 企业知识库文档、版本、权限和引用；
- 推荐反馈、离线评估和 rollout。

发布门：

- 推荐有可解释特征和证据，不只给相似度数字；
- 离线评估使用时间切分，避免未来数据泄漏；
- 未达到足够样本时显示“证据不足”，不展示虚假精度；
- 不同组织的数据和模型特征隔离。

## 8. 跨版本领域边界

| 领域 | 负责 | 不得负责 |
|---|---|---|
| Discovery | 查询、公司/官网/联系人发现、来源命中 | 发送邮件、修改 CRM 阶段 |
| Verification | 邮箱与数据有效性 | 决定销售优先级 |
| Scoring | 画像、维度、规则评分、标签 | 覆盖人工禁止和 suppression |
| Outreach | 账户、模板、活动、发送、事件、回复、抑制 | 管理样品生产 |
| CRM | 线索、商机、任务、询盘、样品、报价 | 抓取外部网站 |
| Monitoring | 快照、变化、信号、预警 | 绕过审批触达客户 |
| Knowledge | 企业能力、案例、证书、受控引用 | 成为业务状态真相来源 |
| Identity & Access | 用户、部门、角色、数据范围 | 承载业务规则 |
| Audit & Compliance | 来源、操作、退订、导出和审计 | 生成业务实体 |

领域之间通过版本化 API、数据库事件和 Outbox 协作。短期保持模块化单体与现有 Worker 架构，只有明确的容量、部署或故障隔离证据才拆分服务。

## 9. 统一演进规则

- 数据库迁移使用 expand -> migrate/backfill -> switch -> contract；
- API、事件和前端兼容只做 additive 变更；
- Prompt、Policy、Schema、Pipeline、Adapter、Template、Sequence 和 Signal Rule 独立版本化；
- 长任务可暂停、恢复、取消、限速、预算控制和幂等重试；
- 所有自动决策保留证据、版本、置信度和人工覆盖；
- 所有危险自动化从 disabled -> shadow -> review -> active，并支持百分比 rollout；
- 回滚保留历史记录，不生成虚构 fallback 结果；
- Redis、消息队列、SSE、飞书和对象存储都不是核心业务状态真相来源；
- 任何外部 URL 输入执行 SSRF 防护；
- 日志、事件、审计和快照不得泄露凭证和不必要 PII。

## 10. 版本治理和发布门

每个子版本必须具备：

1. 用户可完成的端到端结果；
2. 数据模型、状态机、权限和审计；
3. 失败、重试、取消、恢复和部分成功；
4. 配置、凭证、额度和成本；
5. 指标、告警、健康检查和恢复步骤；
6. 空数据库、已有数据库升级、降一版、再升级；
7. 后端测试、静态检查、编译、前端类型检查和构建；
8. 桌面和窄屏视觉检查；
9. disabled/shadow/review/active rollout；
10. 明确的上线中止信号和回滚步骤。

容量目标应作为压测验收目标而不是设计事实。只有 staging 压测、数据库基线和第三方限制共同证明后，才能宣称达到原文档中的规模数字。

## 12. 当前待确认的产品决策

以下决定会实质改变 V2.0 及以后设计，应在进入对应版本前由产品/业务负责人确认：

- Outreach 首批目标国家和适用合规政策；
- 自建 SMTP、企业邮箱 API 还是专业发送 Provider；
- 是否允许自动发送，还是所有首封邮件必须人工批准；
- 组织、部门、品牌和联系人级 suppression 优先级；
- 邮件正文、回复和附件的保留周期；
- CRM 的销售管道、阶段、必填字段和审批金额阈值；
- 样品、报价与现有 ERP/财务系统的边界；
- V3 允许监控的网站范围、许可审核和快照保留策略；
- AI 建议可以自动执行到哪一步，哪些动作永远需要人工确认。

在这些决定确认前，可以完成领域接口、状态和兼容设计，但不得假设权限去发送真实邮件、创建外部审批或批量监控网站。

## 13. 设计门结论

- 产品：通过。版本按用户闭环拆分，近期聚焦持续数据质量而非功能扩张。
- 架构：通过。Discovery、Verification、Scoring、Outreach、CRM、Monitoring 边界明确。
- 数据：通过设计门。各版本使用 additive migration、历史版本和来源证据。
- 可靠性：通过设计门。长任务、发送、事件和监控均要求幂等、Lease、恢复和预算。
- 演进：通过。规则、Prompt、模板、序列、信号和 Adapter 独立版本化。
- 运维：通过设计门。每个子版本必须带指标、告警、rollout 和回滚。
- 安全：通过设计门。抑制优先、凭证治理、SSRF、租户隔离和审计为硬边界。
- 体验：通过设计门。默认减少操作，复杂能力渐进披露，部分结果可用。
- 验证：待各子版本实施证明。本文是路线图，不代表后续功能已经实现。
