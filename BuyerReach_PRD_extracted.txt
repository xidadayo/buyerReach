第一部分 项目背景与产品定义
1. 项目背景
公司需要持续寻找海外品牌、零售商、进口商及其买手、采购、产品开发和供应链联系人。目前依赖人工搜索、Excel记录和分散的第三方工具，存在数据重复、邮箱准确率不清晰、来源不可追溯、人员协作困难及供应商锁定等问题。
人工逐个搜索品牌官网和联系人，效率低。
同一品牌通常存在多名有效联系人，现有表格难以表达一对多关系。
邮箱来源、验证时间、验证服务及可信度无法统一记录。
联系人离职、官网变更、邮箱失效后缺少更新机制。
第三方平台数据难迁移，后续增加开发信、CRM和AI分析时容易推倒重建。
2. 产品定位
BuyerReach是一套部署在公司自有基础设施上的品牌客户发现与开发数据平台。系统通过品牌关键词、品牌名单、国家、品类和职位条件发现目标公司，确认官方网站，获取多个品牌联系人及公开邮箱，执行邮箱验证与评分，并将结果沉淀为可追溯、可去重、可分配、可导出的客户邮箱池。

4. V1成功标准
用户能够创建搜索任务并指定品牌关键词、国家、品类、目标职位和联系人数量。
系统能够保存品牌、公司、官网、官网公开邮箱、多个联系人及其邮箱。
邮箱经过格式、域名、MX及第三方验证后进入对应状态池。
重复品牌、联系人和邮箱能够自动识别并进入合并流程。
管理员可以配置数据源API、规则、任务额度和验证优先级。
业务人员可以筛选、审核、分配、标记和导出有效联系人。
系统能够在另一台支持Docker的服务器上恢复运行。
5. V1非目标
不实现绕过登录、验证码或平台限制的LinkedIn批量爬取。
不实现高并发冷邮件群发和邮箱预热。
不实现完整报价、样品、订单和销售预测CRM。
不承诺邮箱验证100%准确；系统输出可信等级和风险状态。
不以n8n、飞书多维表格或任何SaaS作为核心数据库。

12. 品牌库与公司库
品牌与公司主体分开存储，一个公司可拥有多个品牌。
保存品牌标准名称、别名、官网、国家、城市、类别、公司类型、社媒链接、来源和更新时间。
官网支持多个候选域名，但仅一个主官网；每个候选域名记录可信度和确认状态。
品牌详情页展示官网邮箱、联系人列表、来源证据、标签、负责人和开发状态。
支持手动新建、批量导入、编辑、归档、合并和重新丰富。
13. 官网发现与解析
通过搜索数据源返回官网候选，结合品牌名称、公司名称、国家、页面内容和社媒交叉判断。
扫描首页、About、Contact、Team、Wholesale、Careers、Privacy、Terms、Press及Footer。
提取公开邮箱、电话、地址、公司名称、社媒链接和联系表单URL。
保存来源URL、抓取时间、页面标题、证据摘要和内容哈希。
遵守robots、访问频率和超时规则；失败任务可重试，不得无限循环。
14. 联系人发现
按职位词库和职位优先级查找联系人。
每个品牌默认最多保存5名目标联系人，但数据模型不设硬性一对一限制。
保存姓名、职位、部门、所在国家、公司、公开资料链接、来源、在职状态和最后确认时间。
联系人来源可以是商业数据API、公开网页、搜索结果或人工导入。
LinkedIn链接仅作为资料字段和人工核实入口，V1不自动登录或批量抓取。

15. 邮箱发现
支持公开邮箱提取、商业数据服务返回、已知邮箱模式推断和人工录入。
公开邮箱需记录具体来源URL；第三方返回邮箱需记录服务商和响应时间。
推断邮箱必须保留推断规则，如firstname.lastname@domain。
部门邮箱和个人邮箱分开分类。
一个联系人可关联多个邮箱，并标记主邮箱。

17. 邮箱池
原始数据池：未经验证的候选邮箱。
待验证池：已排队等待验证。
人工审核池：Catch-all、来源冲突、职位不确定等。
有效邮箱池：达到系统可用条件的邮箱。
无效邮箱池：明确无效或一次性邮箱。
禁止发送池：退订、投诉、硬退信和人工禁止记录，优先级最高。
18. 人工审核中心
审核官网候选、联系人归属、职位有效性、风险邮箱和重复数据。
支持批准、驳回、修改、合并、标记待观察和加入黑名单。
所有审核操作记录审核人、时间、原值、新值和原因。
支持批量审核，但加入禁止发送池等高风险操作需二次确认。

20. 导入导出
支持CSV、XLSX导入品牌、域名、联系人和邮箱。
导入前提供字段映射、预览、错误行下载和重复策略。
支持按筛选条件导出品牌、联系人、邮箱、来源和验证状态。
导出完整邮箱需具备专门权限并记录审计日志。
提供JSON和REST API用于未来系统集成。
21. 标签、自定义字段与配置中心
品牌、联系人和邮箱支持手动标签与规则自动标签。
管理员可创建文本、数字、日期、单选、多选、布尔、网址、邮箱、电话等自定义字段。
职位词库、排除词、邮箱评分规则、数据源优先级、重试规则和任务并发必须配置化。
V1可实现基础自定义字段，不要求复杂计算字段和跨对象公式。
22. 用户、权限与审计
采用RBAC角色权限和数据范围权限。
支持本人、部门和全部数据三个基础范围。
API密钥、完整邮箱导出、删除、合并、黑名单和系统配置为敏感权限。
记录登录、查看敏感信息、导出、修改、删除、合并和权限变更日志。
23. 飞书与通知
V1支持通过Webhook向飞书发送任务完成、任务失败、API额度不足和待审核数量提醒。
通知模块使用统一Provider接口，后续可以接入企业微信、邮件或Slack。
飞书不作为核心数据库，不要求V1实现双向多维表格同步。

26. 通用字段规范
主键统一使用UUID。
所有业务表包含created_at、updated_at、created_by、updated_by。
需要软删除的表包含deleted_at和deleted_by，禁止直接物理删除核心业务数据。
组织隔离表预留organization_id；业务归属预留owner_id和department_id。
外部系统记录provider、external_id、raw_payload_hash和last_synced_at。
时间统一在数据库保存UTC，前端按用户时区显示。
27. 数据来源模型
每个品牌、官网、联系人和邮箱可以关联多个来源证据。
来源类型包括official_website、search_engine、commercial_api、public_directory、manual_import、manual_entry。
来源保存URL、标题、摘要、抓取时间、服务商和可信等级。
不得只保存最终值而丢弃数据来源。

38. 数据库开发规范
表名和字段名使用snake_case英文。
所有外键建立索引；高频筛选字段建立组合索引。
邮箱保存标准化小写值，并对organization_id+normalized_address建立唯一约束。
根域名保存标准化值，并考虑国际化域名转Punycode。
JSONB仅用于第三方原始字段、动态筛选条件和非核心扩展信息，不替代规范化关系表。
事务由Service层控制；跨模块更新应使用同一事务或事件最终一致性。
生产环境禁止自动create_all，必须使用Alembic升级。
39. 异步任务规范
任务必须幂等：同一task_item重复执行不得产生重复核心记录。
外部API失败采用指数退避，默认最多3次；认证失败和额度耗尽不自动重试。
每个任务记录started_at、finished_at、attempts、provider、error_code和error_message。
任务支持超时、取消和死信处理。
批量任务按品牌或域名拆分为可独立重试的task_item。
Celery Beat负责定期重验和数据质量任务，n8n不承担高频队列职责。
40. 前端开发规范
使用TypeScript严格模式，禁止核心业务使用any。
API调用统一封装，自动处理Token、request_id、错误提示和取消请求。
列表页统一支持分页、排序、筛选、保存筛选和列显示配置。
所有状态颜色和标签由统一枚举映射，不在页面内散落硬编码。
敏感邮箱默认可根据权限局部脱敏。
批量删除、合并、导出和黑名单操作必须二次确认。
任务进度通过轮询或SSE/WebSocket显示；V1可优先轮询。
界面文字通过i18n文件管理，为英文后台预留。

第七部分 安全、合规与权限
44. 身份认证
V1采用用户名/邮箱+密码登录和JWT短期访问令牌。
密码使用Argon2或bcrypt安全哈希。
连续登录失败进行临时锁定或限流。
管理员可禁用账号和强制重置密码。
后续预留SSO和飞书身份登录接口。
45. 数据安全
生产环境必须启用HTTPS。
数据库、MinIO和Redis不得直接暴露公网。
API密钥、Webhook地址和必要敏感配置加密保存。
备份文件加密并设置保留周期。
导出文件设置有效期，下载需鉴权。
删除采用软删除；真正物理清理需单独维护任务和审批。
46. 数据合规与来源追溯
系统只处理业务开发所需的公司和职业联系信息。
保存数据来源、获取日期、使用目的、验证记录和删除/退订记录。
禁止发送池对所有后续导入和采集具有最高优先级。
不以自动化方式绕过登录、验证码、访问控制或平台条款。
开发信模块上线前应另行制定目标市场适用的隐私、退订和邮件发送政策。

58. 跨版本统一开发原则
所有新模块必须通过稳定的领域接口访问品牌、公司、联系人和邮箱数据，不得直接绕过业务层修改核心表。
每个版本的数据库变更必须包含 Alembic 迁移、回滚说明和升级前备份检查。
所有第三方服务继续使用 Provider/Adapter 机制，禁止将供应商特有字段扩散到核心领域模型。
新增自动化功能必须提供人工暂停、人工审核、失败重试、额度限制和审计日志。
任何发送、导出、删除、合并、退订和黑名单操作必须受权限控制并记录操作人。
各版本新增页面必须兼容中文界面和未来英文国际化，界面文字不得写死。
新增任务必须进入统一任务中心，记录进度、结果、消耗、错误和重试历史。
版本升级不能破坏 V1.0 的 Excel/CSV/JSON 导出能力和 Docker 可移植部署能力。

配置监控范围 → 定时抓取公开页面 → 内容标准化 → 与上次快照比较 → 生成变化事件 → 信号分类和评分 → 去重 → AI摘要 → 人工审核 → 创建客户任务或商机

62.6 AI治理要求
AI推荐必须附带可查看的公开证据，不允许只给出无法解释的结论。
信号评分应区分规则分、模型分和人工修正分。
模型和提示词升级后保留版本，并支持对历史样本进行回归评估。
高风险动作（自动创建活动、自动发送）默认关闭，须经管理员开启并设置审批。
知识库内容必须有版本、有效期、负责人和适用范围。

62.8 V3.0验收标准
系统能够对已配置的官网页面生成快照并识别真实内容变化，重复内容不重复告警。
采购信号包含类型、时间、证据、评分、置信度和处理状态。
管理员可配置哪些信号组合达到阈值后通知业务员或创建待审核线索。
相似客户推荐能够说明相似维度，并可排除已开发、已拒绝或不符合客户。
AI开发建议能够结合公司知识库、品牌变化和联系人职位输出建议，但默认不自动发送。
管理驾驶舱能够展示从信号发现到回复、商机和成交的完整漏斗。
62.9 V3.0不包含
绕过网站访问控制的抓取

未经审核的全自动大规模发送
替代ERP生产和财务系统
无证据的自动客户决策

65.2 可用性
后台核心业务月可用性目标不低于99.5%。
第三方Provider不可用时，任务应进入重试或备用Provider，不影响已保存数据。
任务、发送和Webhook消费者支持水平扩展。
数据库、MinIO和配置文件应有自动备份及恢复演练。
65.3 安全
API密钥、SMTP凭据和Webhook密钥使用加密存储。
所有外部Webhook必须验证签名或共享密钥。
导出邮箱、查看完整邮箱和批量发送纳入高敏权限。
支持登录失败限制、会话失效、操作审计和可选双因素认证。
敏感正文和凭据不得写入普通应用日志。
65.4 可观测性
统一结构化日志并包含 trace_id、task_id、user_id 和 provider。
统计任务成功率、队列积压、Provider错误率、发送事件延迟和成本。
关键错误通过飞书或邮件告警。
每个版本发布后配置核心业务健康检查和Grafana看板。
66. 版本发布与兼容规范
采用语义化版本：主版本.次版本.修订版本，例如 2.0.0。
API 默认使用 /api/v1 路径；破坏性变更必须新增API版本并提供迁移周期。
数据库迁移必须先在测试环境使用生产数据脱敏副本验证。
新功能使用 Feature Flag 分阶段启用，涉及发送、AI自动化和监控的功能默认关闭。
发布包必须包含 changelog、迁移说明、配置差异、回滚步骤和验收清单。
旧字段废弃必须先标记 deprecated，至少跨一个次版本后才能删除。
Provider响应应保存原始结果或可追溯摘要，适配器升级不得破坏历史数据。

69. 后续版本优先级建议
建议严格按照“数据质量优先、发送安全其次、CRM承接回复、最后做主动监控”的顺序推进。若V1.0上线后业务团队急需开发信，可在V1.5中优先完成客户评分、抑制名单和数据复查，再进入V2.0最小发送闭环，不能直接跳过邮箱质量和停止规则。
